\section{Stakeholder Analysis}
\label{section:stakeholder-analysis}

Stakeholders are individuals, groups, or entities that have an interest in, or
are affected by, the outcomes of a project. The plugin involves
five distinct stakeholder groups, each with different relationships to the
system and different expectations of its behaviour.

\textbf{Game Developer (Direct User).} The game developer is the primary
technical user of the plugin. This stakeholder integrates the plugin into an
Unreal Engine 5 project, defines which blend shape and morph target parameters
the plugin may control, and selects the LLM operating mode — local inference via
llama.cpp or cloud fallback through a supported provider. The game developer is
responsible for exposing the plugin's character creation widget within the game's
user interface and for configuring any API credentials required for cloud
operation. Their principal concern is that the plugin integrates cleanly as a
drop-in component without requiring modifications to existing skeletal mesh
assets or animation blueprints.

\textbf{Player (Indirect User).} The player interacts with the system
exclusively through the natural-language character creation interface presented
at runtime. This stakeholder has no awareness of the underlying LLM or parameter
mapping mechanism; they expect to describe a character in plain text and receive
a visually faithful result on the skeletal mesh. The player's primary concern is
expressiveness — that the system correctly interprets descriptive terms relating
to facial structure, proportions, and appearance — and responsiveness, in that
the generation latency does not disrupt the flow of gameplay.

\textbf{Project Team.} The project is developed by two core developers
responsible for the plugin architecture, LLM integration, and parameter mapping
pipeline. A third team member contributes to dataset curation and evaluation,
ensuring that the prompt--parameter mapping is validated against a representative
range of character descriptions and skeletal mesh configurations. The team's
collective interest is in delivering a system that is technically sound,
reproducible, and suitable for academic assessment.

\textbf{Academic Advisors.} The primary academic advisor provides guidance on
research methodology, system design, and evaluation criteria. A co-advisor is
yet to be confirmed. The advisors' interest lies in ensuring the project meets
the academic rigour required for a senior project submission, including clear
problem framing, traceable requirements, and a principled evaluation of the
system's performance against its stated objectives.

\section{User Stories and Use Cases}
\label{section:user-stories-use-cases}

The following user stories capture the needs of each stakeholder group with
respect to the system.

\subsection{Game Developer.}
\begin{enumerate}[label=\textbf{US-D-\arabic*.}, leftmargin=*, align=left]
    \item As a game developer, I want to register my skeletal mesh's blend shape names through a configuration asset so that the plugin knows which parameters to
    populate without requiring changes to my existing assets.
    \item As a game developer, I want to select between local and cloud LLM inference at
    setup time so that I can choose the mode appropriate for my project's
    connectivity and quality requirements.
    \item As a game developer, I want the plugin to fire a Blueprint event containing the
    parsed parameter values so that I can handle how those values are applied within
    my own game logic.
    \item As a game developer, I want the plugin to fall back automatically to an
    alternative LLM or default values when the primary mode fails so that character
    creation does not block the player.
    \item As a game developer, I want to select a preset configuration 
    for popular character systems so that I can reduce initial setup 
    time without manually registering every parameter from scratch.
    
    \item As a game developer, I want to provide my API key through 
    a configuration file so that cloud LLM services can be accessed 
    securely without hardcoding credentials into my project.
    
    \item As a game developer, I want the plugin to fire a Blueprint 
    event when generation fails so that I can handle the error 
    gracefully within my own game logic and inform the player 
    accordingly.
\end{enumerate}

\subsection{Player.}
\begin{enumerate}[label=\textbf{US-P-\arabic*.}, leftmargin=*, align=left]
    \item As a player, I want to type a plain-text description of my character so that I
    do not have to manually adjust individual sliders or understand abstract
    parameter names.
    \item As a player, I want the generated character to visually reflect the description
    I provided so that the result matches my intent.
    \item As a player, I want character generation to complete within a reasonable time so
    that it does not disrupt the flow of starting the game.
    \item As a player, I want to see a visual indicator while my 
    character is being generated so that I am aware the system is 
    processing my input and has not frozen.
    \item As a player, I want to be able to regenerate my character 
    with a new description if the result does not match my intent 
    so that I can refine my character before starting the game.
\end{enumerate}

\subsection*{Use Cases}

\textbf{UC-01 (Brief): Register Character Parameters.}
The game developer creates a Data Asset in their Unreal Engine project, lists
the semantic parameter names and their corresponding blend shape identifiers,
and saves the asset. The plugin reads this configuration at runtime to determine
which parameters it may populate.

\textbf{UC-02 (Casual): Automatic LLM Fallback.}
The plugin attempts to send the player's description to the local llama.cpp
inference server. If the server is unreachable or returns a malformed response,
the plugin reattempts the request against the configured cloud provider — GPT-4o
mini, Claude Haiku, or Gemini Flash, in priority order. If all providers fail,
the plugin fires the Blueprint event with default parameter values so that
character creation is not blocked. The game developer is notified of the failure
mode through an Unreal Engine log warning.

4. The LLM returns a JSON object mapping each registered parameter to a
numerical value between 0 and 1. \newline
5. The plugin validates the response structure. \newline
6. The plugin fires a Blueprint event with the parsed parameter map. \newline
